# bit collaboration protocol v1

## 1. Purpose

This document defines v1 of the internal `Hub` persistence format used by `bit pr` / `bit issue`, along with the merge gate contract.
It covers the following:

- Hub record key/kind/payload contract
- PR merge policy (`.git/hub/policy.toml`) contract
- Workflow result integration (`bit pr workflow`, `workspace flow --pr`) contract
- Authentication and signing policy inputs for relay sync

## 2. Persistence

- Hub records are managed under `refs/notes/bit-hub`.
- Records are serialized as `HubRecord`.
- Workflow records use the following key space:
  - prefix: `hub/pr/<pr_id>/workflow/`
  - key: `hub/pr/<pr_id>/workflow/<task_hash>`
  - kind: `hub.pr.workflow`

`<task_hash>` is a stable hash computed from the task name; the same task overwrites the same key.

## 3. workflow payload (`hub.pr.workflow`)

The payload is a JSON object:

```json
{
  "pr_id": "p-123",
  "task": "test",
  "status": "success",
  "workspace_fingerprint": "...",
  "author": "ci@example.com",
  "txn_id": "flow-test-...",
  "summary": "optional"
}
```

- `status`: `pending|running|success|failed`
- `workspace_fingerprint`: workspace fingerprint at execution time (string)
- `txn_id`: optional
- `summary`: optional

`updated_at` uses the HubRecord timestamp as the source of truth, not the payload.

## 4. merge policy v1

`.git/hub/policy.toml` generated by `bit pr init` (or `bit issue init`):

```toml
[merge]
required_approvals = 0
allow_request_changes = true
require_signed_records = false
required_workflows = []
```

When `required_workflows` contains task names, `bit pr merge` must satisfy the following:

- A workflow record exists for each task on the target PR
- The latest status for each task is `success`

If not satisfied, the merge is rejected.

## 5. CLI contract

### 5.1 workflow recording

- `bit pr workflow submit <pr-id> --task <task> --status <status> [--fingerprint <v>] [--txn <id>] [--author <name>] [--summary <text>]`
- `bit pr workflow list <pr-id>`

### 5.2 workspace integration

- `bit workspace flow <task> --pr <pr-id>`
  - Automatically records the flow execution result as `hub.pr.workflow`
  - `status`: `success` if all required nodes succeed, `failed` if any fail
  - `txn_id`: workspace flow transaction id

## 6. Authentication and signing inputs for relay sync

`bit relay sync push/fetch` accepts the following options:

- `--auth-token <token>`
- `--signing-key <key>` (`--sign-key` is also accepted)
- `--require-signed`
- `--allow-unsigned`

When not specified, environment variables are used:

- `BIT_RELAY_AUTH_TOKEN`
- `BIT_COLLAB_SIGN_KEY`
- `BIT_COLLAB_REQUIRE_SIGNED`

CLI options take precedence over environment variables.

## 7. Compatibility

- Existing PR/Issue/Comment/Review/Note record formats are unchanged.
- When `required_workflows` is unset (empty array), behavior is the same as the previous merge policy.
