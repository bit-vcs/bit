# bit collaboration protocol

This document defines the persistence and synchronization contract used by
`bit issue` / `bit pr`.

It has two goals:

- document the current v1 behavior already implemented in `bit`
- define the planned v2 direction for a GitHub-like, local-first, P2P-capable
  collaboration layer

The protocol is shared conceptually by `bit`, `bit-relay`, `bithub`, and the
workflow runner layer.

## 1. Purpose and Scope

This document covers the following:

- Hub record key/kind/payload contract
- PR merge policy (`.git/hub/policy.toml`) contract
- workflow result integration (`bit pr workflow`, `workspace flow --pr`)
- source-of-truth boundaries between Git, relay, and runtime blob storage
- planned v2 record families for issue/pr/review/run/check/artifact/sync

The Git smart-http specification itself is out of scope.

## 2. Persistence and Source of Truth

### 2.1 Collaboration metadata

The canonical collaboration metadata store is Git:

- records are managed under `refs/notes/bit-hub`
- each logical record is serialized as a `HubRecord`
- issue/pr/review/comment/work-item metadata is expected to remain cloneable as
  part of repository state

This is the system of record for collaboration state.

### 2.2 Relay

`bit-relay` is not the canonical source of truth for collaboration metadata.
Its responsibilities are:

- mutation transport
- notify / subscribe transport
- sync cursor management
- replication / fan-out / P2P coordination

Relay may cache or index records, but the canonical persisted collaboration
state remains Git-backed.

### 2.3 Runtime execution data

Workflow runtime data is split as follows:

- Git stores durable summaries needed for review and merge decisions
- blob/object storage stores large runtime data

Canonical blob/object storage data includes:

- full run logs
- step logs
- artifacts
- cache payloads

Git should keep only stable summaries and references needed by CLI/UI/merge
gates.

## 3. Current v1 Contract

This section describes behavior that already exists today.

### 3.1 Persistence

- Hub records are managed under `refs/notes/bit-hub`.
- Records are serialized as `HubRecord`.
- Workflow records use the following key space:
  - prefix: `hub/pr/<pr_id>/workflow/`
  - key: `hub/pr/<pr_id>/workflow/<task_hash>`
  - kind: `hub.pr.workflow`

`<task_hash>` is a stable hash computed from the task name; the same task
overwrites the same key.

### 3.2 Workflow payload (`hub.pr.workflow`)

The payload is a JSON object:

```json
{
  "pr_id": "p-123",
  "task": "test",
  "status": "success",
  "workspace_fingerprint": "...",
  "author": "ci@example.com",
  "txn_id": "flow-test-...",
  "summary": "optional"
}
```

- `status`: `pending|running|success|failed`
- `workspace_fingerprint`: workspace fingerprint at execution time (string)
- `txn_id`: optional
- `summary`: optional

`updated_at` uses the `HubRecord` timestamp as the source of truth, not the
payload.

### 3.3 Merge policy v1

`.git/hub/policy.toml` generated by `bit pr init` (or `bit issue init`):

```toml
[merge]
required_approvals = 0
allow_request_changes = true
require_signed_records = false
required_workflows = []
```

When `required_workflows` contains task names, `bit pr merge` must satisfy the
following:

- a workflow record exists for each task on the target PR
- the latest status for each task is `success`

If not satisfied, the merge is rejected.

### 3.4 CLI contract

#### Workflow recording

- `bit pr workflow submit <pr-id> --task <task> --status <status> [--fingerprint <v>] [--txn <id>] [--author <name>] [--summary <text>]`
- `bit pr workflow list <pr-id>`

#### Workspace integration

- `bit workspace flow <task> --pr <pr-id>`
  - automatically records the flow execution result as `hub.pr.workflow`
  - `status`: `success` if all required nodes succeed, `failed` if any fail
  - `txn_id`: workspace flow transaction id

## 4. Planned v2 Direction

This section is a draft. It defines the intended contract for the generalized
task/collaboration layer.

### 4.1 Design goals

- keep GitHub terminology at the CLI/API/UI layer
- support local-first and P2P workflows without requiring GitHub
- allow GitHub sync through adapters rather than hard-coding GitHub into the
  core schema
- keep repository-cloneable metadata in Git
- keep large runtime payloads outside Git
- avoid exposing schema version details in end-user commands

### 4.2 External terminology and internal model

User-facing concepts remain GitHub-like:

- Issue
- Pull Request
- Review
- Workflow Run
- Check
- Artifact

The internal canonical model is centered on a generalized work-item record plus
specialized extension records.

### 4.3 Common envelope fields

All planned v2 record families should support these logical fields:

- `id`
- `version`
- `kind`
- `created_at`
- `updated_at`
- `author`
- `source`
  - `local | relay | github`
- `sender`
- `device_id`
- `room`
- `signature`
- `sync_state`
  - `local_only | pending_push | synced | conflict`
- `visibility`
  - `repo | room | private`
- `upstream`
  - `provider`
  - `repo`
  - `remote_id`
  - `remote_number`
  - `remote_url`

These fields are intentionally generic enough to support future providers and
P2P identity expansion.

### 4.4 Planned record families

#### `hub.work-item.v2`

Canonical shared record for both issues and pull requests.

Fields:

- `id`
- `kind`
  - `issue | pull_request`
- `title`
- `body`
- `state`
  - issues: `open | closed`
  - pull requests: `open | closed | merged | draft`
- `labels[]`
- `assignees[]`
- `linked_prs[]`
- `parent_id`
- `patch`
  - `source_branch`
  - `source_repo`
  - `source_ref`
  - `source_commit`
  - `target_branch`
  - `target_commit`
  - `closes_issues[]`
  - `merge_commit`
- `workflow_summary`
  - `latest_run_status`
  - `required_checks[]`
  - `successful_checks[]`

The current `WorkItem` / `Issue` / `PullRequest` layout already points in this
direction and should remain the compatibility anchor.

#### `hub.issue.comment.v2`

Issue comment record.

Fields:

- `comment_id`
- `issue_id`
- `author`
- `body`
- `created_at`
- `reply_to`

Threading remains explicit through `reply_to`.

#### `hub.pr.review.v2`

Pull request review record.

Fields:

- `review_id`
- `pr_id`
- `verdict`
  - `comment | approve | request_changes`
- `body`
- `commit_id`

Inline review comments may remain a later extension record if needed.

#### `hub.run.summary.v2`

Workflow run summary record stored in Git.

Fields:

- `run_id`
- `work_item_id`
- `workflow_name`
- `trigger`
- `executor`
  - `local | container | cloudflare_sandbox`
- `status`
  - `queued | running | success | failed | cancelled`
- `started_at`
- `finished_at`
- `summary`
- `check_ids[]`
- `artifact_ids[]`
- `log_ref`

This record is intended to be merge-gate friendly and UI-friendly.

#### `hub.check.v2`

Check/job/step summary record stored in Git.

Fields:

- `check_id`
- `run_id`
- `work_item_id`
- `name`
- `status`
  - `queued | running | success | failed | skipped`
- `summary`
- `started_at`
- `finished_at`
- `log_ref`

#### `hub.artifact.ref.v2`

Artifact reference record stored in Git, with large payloads stored elsewhere.

Fields:

- `artifact_id`
- `run_id`
- `work_item_id`
- `name`
- `content_type`
- `size`
- `digest`
- `blob_url` or `object_key`
- `retention`

#### `hub.sync.link.v2`

External sync mapping record.

Fields:

- `local_id`
- `local_kind`
- `provider`
- `repo`
- `remote_id`
- `remote_number`
- `remote_url`
- `last_pull_at`
- `last_push_at`
- `sync_policy`

This record is the main bridge for GitHub issue/pr synchronization.

## 5. Relay and Sync Considerations

### 5.1 Relay role

Relay should transport mutations and notifications without becoming the primary
store for collaboration truth.

Intended relay responsibilities:

- publish/apply collaboration mutations
- provide cursors and replay windows
- support P2P synchronization
- carry signatures and sender/device metadata

### 5.2 Upstream adapters

GitHub integration should be implemented as an adapter layer.

The core collaboration model must not require GitHub-specific identifiers, but
it should preserve enough mapping information to support:

- import
- pull sync
- push sync
- webhook-triggered updates
- dry-run / conflict inspection

## 6. Authentication and Signing Inputs for Relay Sync

`bit relay sync push/fetch` accepts the following options:

- `--auth-token <token>`
- `--signing-key <key>` (`--sign-key` is also accepted)
- `--require-signed`
- `--allow-unsigned`

When not specified, environment variables are used:

- `BIT_RELAY_AUTH_TOKEN`
- `BIT_COLLAB_SIGN_KEY`
- `BIT_COLLAB_REQUIRE_SIGNED`

CLI options take precedence over environment variables.

## 7. Compatibility and Migration

- existing PR/Issue/Comment/Review/Note records remain readable
- existing work-item records remain the compatibility base
- current CLI commands should stay schema-version agnostic
- read-time upgrade is preferred over mandatory eager migration
- downgrade/export paths may be added for interoperability if needed
- when `required_workflows` is unset (empty array), merge behavior stays the
  same as v1

## 8. Open Questions

- should draft PR be represented directly in `state` or as a separate field?
- should milestone/project concepts be part of core v2?
- should reactions, attachments, and inline review comments be core records or
  extension records?
- should upstream conflict resolution be `last_write_wins`, field-wise merge,
  or configurable per provider?
